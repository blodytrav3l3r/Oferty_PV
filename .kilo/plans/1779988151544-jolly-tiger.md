# Plan naprawy modułu studni — nawigacja kreatora

## Analiza problemów

### 1. Krok 1 — "Dalej" na dolnej belce nie działa

**Przyczyna**: Przycisk "Dalej" w bottom nav (`.wizard-nav-fixed`) ma dwie definicje onclick:
- Statyczna w HTML: `onclick="wizardNext()"` 
- Dynamiczna w `updateStudnieBottomNav()`: `nextBtn.onclick = wizardNext`

Obie wskazują na `wizardNext()` → `wizardNavStep(2)` → `goToWizardStep(2)`. Kod jest poprawny, ale **bottom nav jest OUTSIDE `.section-builder`** (poprzednia poprawka). Jeśli `showSection()` przełącza widoczność sekcji, nav pozostaje widoczny — to jest OK.

**Prawdziwa przyczyna**: `updateStudnieBottomNav()` jest wywoływane z `goToWizardStep()` które jest wywoływane przy inicjalizacji. Jeśli `globals.js` (definiuje `currentWizardStep = 1`) nie został załadowany PRZED `uiHelpers.js`, zmienna `currentWizardStep` jest `undefined`, co powoduje błąd w `wizardNavStep(undefined + 1 = NaN)`.

**Sprawdź**: Kolejność ładowania skryptów w `studnie.html`.

### 2. Krok 2 — "Dalej" nie aktywuje się po wybraniu parametrów

**Przyczyna**: `validateWizardStep2()` kontroluje `studnie-nav-next` (bottom nav). Walidacja wymaga:
- Wszystkie 18 parametrów z `WIZARD_REQUIRED_PARAMS` potwierdzone w `wizardConfirmedParams` (Set)
- Jeśli malowanie wybrane (`malowanieW`/`malowanieZ` !== 'brak'), pole nazwy powłoki i ceny muszą być wypełnione

`wizardConfirmedParams` jest aktualizowane przez event listener na kafelkach parametrów. `validateWizardStep2()` jest wywoływane:
- Przy wejściu na krok 2 (`goToWizardStep(2)` → linia 62)
- Po każdej zmianie kafelka (w `wellUI.js` lub `wellManager.js`)

**Problem 1**: Jeśli `validateWizardStep2()` nie jest wywoływane po kliknięciu kafelka (event nie bubble'uje lub handler nie istnieje), przycisk pozostaje disabled.

**Problem 2**: `WIZARD_REQUIRED_PARAMS` zawiera 18 pozycji. Wszystkie muszą być w `wizardConfirmedParams`. Jeśli któryś parametr nie ma kafelka w DOM lub nie rejestruje się w secie, walidacja nigdy nie przejdzie.

**Problem 3**: Wewnątrz kroku 2 jest INNY przycisk "Dalej" (`#wizard-next-step2`, linia 1545-1552 studnie.html) z `disabled` w HTML i NIE jest kontrolowany przez `validateWizardStep2()` — pozostaje wiecznie disabled.

### 3. Krok 3 — "Zastosuj do wszystkich" do usunięcia

`updateStudnieBottomNav()` (linia 288) ustawia: `applyBtn.style.display = step === 3 ? 'flex' : 'none'`. Przycisk pojawia się tylko w kroku 3.

**Rozwiązanie**: Usunąć przycisk z bottom nav HTML i usunąć logikę wyświetlania w JS.

### 4. Krok 3 — "Zakończ" nie zapisuje oferty

"Zakończ" w kroku 3 wywołuje `wizardNext()` → `wizardNavStep(4)` → `goToWizardStep(4)`. **Nie zapisuje oferty, nie waliduje danych kroku 2.**

**Oczekiwane zachowanie**: Po kliknięciu "Zakończ":
1. Walidacja parametrów (czy krok 2 jest kompletny)
2. Zapisanie oferty przez `saveOfferStudnie()`
3. Przejście do zakładki "Oferta"

### Krok 1: Fix "Dalej" w kroku 1

**Sprawdzić**: Kolejność ładowania skryptów w `<head>` studnie.html — `globals.js` musi być przed `uiHelpers.js`.

**Jeśli kolejność jest OK**: Problem może być w braku wywołania `updateStudnieBottomNav()` po załadowaniu strony. `goToWizardStep(1)` jest wywoływane przy inicjalizacji? Sprawdzić czy `currentWizardStep = 1` i `goToWizardStep(1)` są wywoływane przy starcie.

**Fix**: Dodać jawne wywołanie `goToWizardStep(currentWizardStep)` lub `updateStudnieBottomNav()` po załadowaniu DOM, jeśli jeszcze nie istnieje.

### Krok 2: Fix "Dalej" w kroku 2

**Sprawdzić**: 
1. Czy `validateWizardStep2()` jest wywoływane po kliknięciu kafelka parametru
2. Czy `wizardConfirmedParams` zawiera wszystkie 18 wymaganych parametrów po wybraniu
3. Czy `getActiveTileValue()` i `.param-tile.active` działają poprawnie

**Fix A**: Dodać `validateWizardStep2()` do event handlera zmiany kafelka (jeśli brak).

**Fix B**: Wyłączyć (disabel) przycisk `#wizard-next-step2` (inline w step 2) — niepotrzebny, bo bottom nav już go zastępuje.

### Krok 3: Usunąć "Zastosuj do wszystkich"

### Krok 4: Fix "Zakończ" — zapisz ofertę i przejdź do Oferty

**W `updateStudnieBottomNav()`** (`uiHelpers.js:291-298`), zmienić handler dla kroku 3:

```js
if (nextBtn) {
    if (step === 4) {
        nextBtn.innerHTML = '<i data-lucide="arrow-right"></i> Przejdź do zamówienia';
        nextBtn.onclick = function () { step4NextAction(); };
    } else if (step === 3) {
        nextBtn.innerHTML = '<i data-lucide="check"></i> Zakończ';
        nextBtn.onclick = async function () {
            if (!validateWizardStep2()) {
                showToast('Najpierw uzupełnij wszystkie parametry w kroku 2', 'error');
                return;
            }
            const saved = await saveOfferStudnie();
            if (saved) {
                showSection('offer');
                showToast('Oferta zapisana pomyślnie', 'success');
            }
        };
    } else {
        nextBtn.innerHTML = '<i data-lucide="chevron-right"></i> Dalej';
        nextBtn.onclick = wizardNext;
    }
}
```

### Ryzyka

- `saveOfferStudnie()` może wymagać istnienia studni w ofercie. Jeśli nie ma żadnej studni, zapis może zwrócić błąd.
- `showSection('offer')` — sprawdzić czy zakładka "Oferta" ma id="section-offer" (jest, linia 3178 studnie.html).
- Krok 1 może wymagać wypełnienia podstawowych pól (numer oferty, klient) przed przejściem dalej. Dodać walidację jeśli potrzeba.
