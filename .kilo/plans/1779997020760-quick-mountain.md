## 1. Architektura projektu

### Backend
- **Express.js + TypeScript + Prisma ORM (SQLite)**
- PDF generation: Puppeteer (~300MB z Chromium)
- DOCX generation: docx + docxtemplater
- Sesja: token-based (cookie + DB)
- Serwowane pliki statyczne z `public/`

### Frontend
- **Vanilla JS + iframe-based SPA** (brak frameworka: React, Vue, itd.)
- Brak bundlera/vite/webpack — wszystkie skrypty ładowane przez `<script>`
- Stan globalny przez `window.*` i globalne zmienne
- Ikonki: Lucide (iconsSlim.js 19KB + lucide.min.js 388KB — oba ładowane!)
- Arkusze kalkulacyjne: SheetJS (xlsx.full.min.js 930KB)

### Moduły frontendowe
| Moduł | HTML | JS | CSS |
|---|---|---|---|
| Landing/login | index.html (443 linie, 21KB) | dashboard.js (17KB) | index.css (20KB) |
| SPA shell | app.html (155 linii, 7KB) | router.js (18KB) | spa.css (2KB) |
| Rury (pipes) | rury.html (399 linii, 25KB) | 8 plików, ~138KB | rury.css (22KB) |

## 2. Krytyczne błędy

### 2.1 Nierenderowane ikony Lucide po dynamicznym HTML
**Pliki**: offerItems.js, offerExports.js, pricelistUi.js, transport.js, wellUI.js, wellPopups.js, offerManager.js  
**Opis**: Po każdym `innerHTML = ...` z `<i data-lucide="...">` trzeba wywołać `lucide.createIcons()`. W modułach rury **nigdy** nie jest wywoływane. Ikony wyświetlają się jako puste `<i>`.

### 2.2 Kaskada pełnych przerenderowań (refreshAll)
**Plik**: `wellManager.js:876-922`  
`refreshAll()` wywołuje **19 funkcji renderujących** na każdej zmianie parametru. Każda używa `container.innerHTML = html`, niszcząc całe drzewo DOM. Bezpośrednia przyczyna: brak celowanych aktualizacji DOM.

### 2.3 Rekurencyjne wywołania click wewnątrz handlerów
**Plik**: `wellUI.js:110-163`  
`setupParamTiles()` — handler kliknięcia programowo wywołuje `.click()` na innych tile'ach, co re-triggeruje ten sam handler. Prowadzi do kaskadowych, nieprzewidywalnych przerenderowań.

### 2.4 Konflikt klas `.action-btn`
**Pliki**: `style.css:1457-1472` vs `zlecenia.css:262-277`  
Dwie różne definicje tej samej klasy (kwadratowy przycisk vs tekstowy). W zależności od kolejności ładowania CSS, przyciski na kartotece wyglądają źle.

### 2.5 Brakujące zmienne CSS
| Zmienna | Użycie | Skutek |
|---|---|---|
| `var(--primary)` | `style.css:937,940`, `rury.css:468,470` | `.catalog-tab.active` — przezroczyste tło/border |
| `var(--border)` | 6 selektorów CSS + ~40 inline JS | Niespójne obramowania |
| `var(--accent-light)` | `style.css:1996` | `.offer-card-price` — brak koloru |
| `var(--indigo-500)` | `studnie.css:841,847` | `.fs-dn-tile` — brak border-color |

### 2.6 Podwójny atrybut `class=""` w studnie.html
**Plik**: `studnie.html:202-360+`  
Elementy typu `<div class="ws-item" class="p-2">` — druga wartość klasy jest **całkowicie ignorowana** przez przeglądarkę. Klasy `.p-2`, `.ui-col-5/6/8`, `.ui-text-sec` nigdy nie są aplikowane.

### 2.7 `innerHTML` z danymi użytkownika bez sanityzacji (stored XSS)
| Lokalizacja | Linia | Źródło |
|---|---|---|
| `dashboard.js` | 204-243 | Wszystkie pola użytkownika (`firstName`, `lastName`, itd.) |
| `router.js` | 390 | `displayName` z API |
| `zlecenia.js` | 338-344 | `offerId`, `wellId` w onclick string |
| `pvSalesUi.js` | 178 | `error.message` |

### 2.8 Dead code — `if (false)` bloki
**Pliki**: `offerManager.js:943-957`, `orderManager.js:943-957`  
Dwa bloki kodu (przywracanie `_customPrzejscieRows`) są trwale wyłączone przez `if (false)`. Nigdy nie wykonają się.

### 2.9 Busy-wait w orderManager.js
**Plik**: `orderManager.js:86-90`  
```js
while (isSavingOffer) { await new Promise((r) => setTimeout(r, 200)); }
```
Blokuje micro-task queue — pętla aktywna co 200ms.

### 2.10 `onclick` string przez template literals (XSS)
**Plik**: `zlecenia.js:338-344`  
```js
`AppZlecenia.editOrder('${o.offerId}', '${o.wellId || ''}')`
```
Jeśli `offerId` zawiera apostrof —łamie JavaScript i otwiera XSS.

## 3. Dostępność (Accessibility)

### 3.1 Brak `<label for="">` — wszystkie formularze
- `rury.html:144-213` — 14 pól formularza bez `for`
- `index.html:124-135` — login (username, password)
- `index.html:314-384` — admin panel (wszystkie pola)
- `kartoteka.html:257` — search input

### 3.2 Zero ARIA — cały projekt
- Żadnego `role="dialog"`, `aria-modal` na modalach
- Żadnego `aria-label` na przyciskach ikon
- Żadnego `aria-current="page"` na aktywnej nawigacji
- Żadnego `aria-live` na toastach
- Żadnego `aria-busy` na sekcjach ładujących

### 3.3 Punkty nawigacyjne niedostępne z klawiatury
- `rury.html:97-120` — wizard step dots: brak `tabindex`, `role`, handlerów klawiatury
- `index.html:36` — "Pulpit" jako `<div>` a nie `<a>` — brak `tabindex="0"`, `role="link"`
- `studnie.html` — przyciski well list: `onclick="selectWell(${i})"` bez `role`, `tabindex`

### 3.4 Modal — brak focus trapu i zarządzania fokusem
Wszystkie 4+ implementacji modali (pipe length, discount, history, add product) + `appConfirm`:
- Tab może wyjść poza modal
- Fokus nie wraca do triggera po zamknięciu
- Brak `Escape` handlera w niektórych modalach
- Brak `role="dialog"` lub `aria-modal`

### 3.5 Toast — brak roli alertu
`ui.js:11-39` — `role="alert"` i `aria-live="polite"` nie są ustawione.

### 3.6 Niski kontrast
`rury.html:132` — kolor `#71717a` na ciemnym tle. Współczynnik kontrastu ~4.0:1 przy wymaganym 4.5:1 (WCAG AA dla małego tekstu).

### 3.7 `prefers-reduced-motion` — tylko spa.css implementuje
Wszystkie animacje (`fadeIn`, `shimmer`, `bgShift`, `pulse-draft`, `slideDown`, `logoShimmer`, itd.) nie mają wariantów zredukowanego ruchu.

### 3.8 Brak `:focus-visible` — zero selektorów
Wszystkie focus style używają `:focus` zamiast `:focus-visible`, pokazując ring focus na kliknięciu myszą.

## 4. Wydajność i optymalizacja

### 4.1 Rozmiary bundle (before gzip)

| Plik | Rozmiar | Problem |
|---|---|---|
| xlsx.full.min.js | **930 KB** | Ładowany na każdej stronie, używany tylko przy eksporcie |
| lucide.min.js | **388 KB** | Ładowany razem z iconsSlim.js (19KB) — dublet |
| products_studnie.json | ~450 KB (13,616 linii) | Statyczny JSON bez paginacji |
| Font Inter (7 woff2) | ~213 KB | Wszystkie warianty wagi |
| Łącznie studnie.html JS | ~2.3 MB | 30+ skryptów, ~1.7MB po gzip |

### 4.2 30+ blokujących skryptów (studnie.html)
Skrypty ładowane synchronicznie, bez `defer`/`async`. Blokują parsowanie i renderowanie.

### 4.3 DP solver na głównym wątku
`ringOptimizer.js` — `solveDPRings()` z pętlami O(cap × heights) na głównym wątku. Blokuje UI podczas auto-selekcji.

### 4.4 `innerHTML` wszędzie — ~100+ wystąpień
Każda zmiana stanu = destrukcja + przebudowa całego drzewa DOM. Żadne komponenty nie używają targeted DOM updates, DocumentFragment, ani template cloning.

### 4.5 `transition: all` — 60+ wystąpień
Zmienna `--transition: all 0.25s ...` na `:root` propagowana przez 15+ selektorów. Wymusza sprawdzanie każdej właściwości CSS przy każdej zmianie.

### 4.6 Shimmer animacja — `background-position`
`style.css:1892-1899` — animacja `background-position` powoduje repaint całego elementu na każdej klatce. Zamienić na `transform: translateX()` na pseudo-elimencie.

### 4.7 `studnieProducts.find()` w hot loopach
Dziesiątki wywołań `.find(p => p.id === ...)` w pętli — O(n × m) z ~500 produktami. Proponowane: `Map<id, product>`.

### 4.8 Brak cache'owania
- `router.js:322` — `Date.now()` na każdym przeładowaniu iframe (zapobiega cachingowi)
- `zlecenia.js:84` — `Date.now()` na każdym pobraniu templatki
- Brak `preconnect`/`preload` dla assetów

### 4.9 Auto-refresh co 60s bez sprawdzenia czy dane się zmieniły
`zlecenia.js:167-170`, `pvSalesUi.js:182-186` — setInterval wywołuje pełne przeładowanie danych, nawet jeśli nic się nie zmieniło.

### 4.10 Brak debounce na search
`offerItems.js:14-42` — każde naciśnięcie klawisza wywołuje `products.filter()` na całej tablicy. Brak debounce.

## 5. Bezpieczeństwo

### 5.1 Stored XSS (krytyczne)
- `dashboard.js:204-243` — dane użytkowników wstawiane przez `innerHTML` bez sanityzacji
- `zlecenia.js:338-344` — onclick string przez template literals
- `router.js:390` — `displayName` w `innerHTML`

### 5.2 CSP z `unsafe-inline`
- `script-src 'unsafe-inline'` — zezwala na wszystkie inline skrypty
- Pozwala to na XSS, ale jest wymagane przez obecną architekturę (setki inline `onclick`)

### 5.3 Brak sanityzacji w `innerHTML` HTML stringów
Wszystkie funkcje budujące HTML przez template literals z danymi użytkownika (`${user.name}`, `${offer.id}`) są podatne na XSS.

### 5.4 Twardy redirect na 401 w iframe
`dataService.js:13-15` — `window.location.href = 'index.html'` przy 401. W iframe może redirectować parenta zamiast tylko iframe.

### 5.5 Brak CSP, X-Frame-Options
- Brak `Content-Security-Policy` meta
- Brak `X-Frame-Options` (iframe'y są używane celowo, ale bez ograniczeń)

## 6. CSS — problemy i niespójności

### 6.1 Inline style — dominujący wzorzec w HTML

| Plik | Liczba inline style |
|---|---|
| studnie.html | ~300+ (90% elementów) |
| index.html | ~50+ (admin panel) |
| kartoteka.html | ~40+ (80% widocznego HTML) |
| rury.html | 32 |
| app.html | 15+ |

**Skutek**: niemożność utrzymania spójnego designu, brak korzyści z CSS custom properties.

### 6.2 Selektory oparte na inline style (bardzo kruche)
```css
.style.css:1747 — .kartoteka-filter-bar > div[style*="flex: 1"]
studnie.css:1836-1852 — .wizard-form-grid[style*="grid-template-columns: 1fr 1fr"]
rury.css:974-977 — to samo
spa.css:21-23 — .spa-module-iframe[style*='display: block']
```
Przestają działać jeśli zmieni się format inline style (np. spacja, kolejność).

### 6.3 Duplikacja wizard classes — studnie.css + rury.css
~30 klas wizarda (~350 linii) zdefiniowanych osobno w `studnie.css` i `rury.css`. Różnią się tylko prefixem (`.wizard-` vs `.rury-wizard-`). Powinny być wspólne w `style.css`.

### 6.4 Dead CSS selectors
```css
.offer-card-top, .offer-card-title, .offer-card-meta, .offer-card-price,
.offer-card-actions, .offer-badge-wrap, .offer-no-order, .offer-actions-grouped,
.kartoteka-offer-card, .text-success, .text-danger, .text-accent,
.bg-success-soft, .bg-danger-soft, .bg-warn-soft, .bg-hover, .border-danger-soft
```
Żaden z powyższych nie ma dopasowania w HTML/JS.

### 6.5 Problemy przeglądarkowe
- Brak `scrollbar-width`/`scrollbar-color` (Firefox) w `style.css`
- `::webkit-scrollbar` tylko w Chrome/Safari
- `backdrop-filter` bez `-webkit-` prefiksu w studnie.css, zlecenia.css
- `data:image/svg+xml;utf8` — niepoprawne kodowanie

### 6.6 Brak `color-scheme: dark`
Żadna strona nie definiuje `color-scheme` ani `theme-color`. Dla dark-theme-only applikacji to może powodować problemy z natywnymi kontrolkami formularzy w Windows dark mode.

### 6.7 Brak `touch-action: manipulation`
Żaden przycisk/karta/element interaktywny nie ma tej właściwości. Na urządzeniach mobilnych występuje 300ms opóźnienie tapu.

### 6.8 Brak `overscroll-behavior: contain`
Nie ustawione na modalach — przewijanie w modalu może przewijać stronę w tle.

## 7. Jakość kodu JS

### 7.1 Global namespace pollution
Wszystkie funkcje i stan: globalne `window.*`, `var`, `let` w module scope. Żadnych modułów (ES modules są tylko w pvSalesUi.js). ~200+ punktów zanieczyszczenia globalnej przestrzeni.

### 7.2 Monolityczne pliki JS
| Plik | Linie | Problem |
|---|---|---|
| wellSolver.js (runJsAutoSelection) | ~800 linii | Największa funkcja, 7 zagnieżdżonych funkcji, 6 poziomów zagnieżdżenia |
| orderManager.js | 4625 linii | initKartaBudowyStep4() — 355 linii, powinna być 5+ funkcji |
| offerManager.js | 3246 linii | renderOfferSummaryFooter() — 75 linii z colspan arytmetyką |
| pvSalesUi.js (attachActionListeners) | 338 linii | 10+ par zduplikowanego wiązania eventów |

### 7.3 Zduplikowany kod
- `offerCrud.js` — 3 kopie form field assignment (~60 linii każda)
- `zlecenia.js` — ~95 linii zduplikowanych między `buildZlecenieFromPO` a `buildZlecenieFromPageBlock`
- `pvSalesUi.js:637-975` — 10 par zduplikowanych handlerów eventów
- `studnie.css` + `rury.css` — wizard classes (~350 linii)
- 4+ implementacje overlay/modal creation (różne wzorce)
- `calcPrecoPricing()` + `identifyMainChannelAndTributaries()` — zduplikowana funkcja `getZeroScore`

### 7.4 Wzorce anty-wydajnościowe
- `JSON.parse(JSON.stringify(...))` — 10+ miejsc (zamienić na `structuredClone`)
- `innerHTML = html` — 100+ miejsc (zamiast targeted DOM updates)
- `.find()` na ~500 produktach — dziesiątki razy na render
- `console.warn` w walidatorach AppState — spam produkcyjny
- Brak `structuredClone` lub memoizacji

### 7.5 Potencjalne memory leaki
- SPA router cache'uje iframe ale nigdy nie niszczy
- `setInterval(checkBackendStatus, 15000)` — brak cleanup przy nawigacji
- 60-sekundowy timeout na cleanup iframe w zlecenia.js
- Event listenery na modalach dodawane ale nie usuwane

## Podsumowanie

Projekt cierpi na typowe problemy wieloletniego developmentu bez bundlera i frameworka:

- **Największe problemy wydajnościowe**: brak bundlera (2.3MB JS), `innerHTML` wszędzie (100+), kaskada `refreshAll()` (19 renderów na kliknięcie)
- **Największe problemy bezpieczeństwa**: stored XSS przez `innerHTML` z danymi użytkownika (5+ miejsc)
- **Największe problemy accessibility**: zero ARIA, brak labeli, brak focus managementu
- **Największe problemy utrzymaniowe**: inline style w 90% HTML, monolit orderManager.js (4625 linii), zduplikowany kod (wizard classes, modal creation, field assignment)

**Priorytet**: Faza 1 (krytyczne błędy, 1-2 dni) → Faza 2 (wydajność, 1-2 tygodnie) → Faza 3 (accessibility, 1 tydzień).
