### E. Migracja inline `style=""` → klasy CSS w `studnie.html` — WYSOKI PRIORYTET, WYSOKI WKŁAD

**Plik:** `public/studnie.html` (~3992 linii)

**Problem:** Setki atrybutów `style="..."` rozsianych po całym pliku. To utrudnia utrzymanie i zmiany motywu.

**Zakres:**
- Identyfikacja powtarzalnych wzorców inline styli
- Utworzenie klas CSS w `css/studnie.css` lub `css/style.css`
- Zastąpienie `style="..."` klasami CSS

**Przykładowe wzorce do wyodrębnienia:**
- `style="display: flex; align-items: center; gap: X;"` → `.flex-center`, `.flex-gap-X`
- `style="font-size: Xrem; font-weight: Y; color: Z;"` → `.text-X`, `.fw-Y`
- `style="padding: X; border-radius: Y; background: Z;"` → `.card-X`, `.btn-X`

**Uwaga:** Czasochłonne (~1-2h pracy). Ryzyko niskie, ale wymaga weryfikacji wizualnej po zmianach.

## Ryzyka i uwagi

- **A (kartotekaInit.js):** Funkcje inline są wywoływane przez `onclick` w HTML. Po ekstrakcji muszą pozostać globalne (nie IIFE).
- **D (pricelist.js):** `rury.html` ładuje ten plik — trzeba upewnić się, że nowy shim jest kompatybilny, a dane są dostępne przez `default_products_rury.js`.
- **E (inline style → klasy):** Największy wysiłek. Wymaga przeglądu każdego `style` atrybutu i utworzenia klas CSS. Ryzyko wizualnych różnic.
- **Template'y (`templates/*.html`):** Inline CSS w plikach szablonów jest celowe — są one renderowane przez Puppeteer jako samodzielne dokumenty HTML. Nie ekstrahować.
