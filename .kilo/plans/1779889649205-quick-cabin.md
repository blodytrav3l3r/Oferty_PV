## Plan krok po kroku

### Krok 1: Usuń pliki serwisowe antiGravity

```bash
git rm -r src/services/antiGravity/
```

Usuwane pliki:
- `src/services/antiGravity/index.ts` (315 linii) — orchestrator `selectManholeComponents()`
- `src/services/antiGravity/selection.ts` (196 linii) — funkcje DB `selectDennica()`, `selectKragByDn()`, `selectZakonczenie()`, `selectPlytaRedukcyjna()`, `selectKragiNadbudowy()`, `selectKragOt()`, `getAvailableComponents()`
- `src/services/antiGravity/passages.ts` (235 linii) — logika przejść `placePrzejscia()`, `insertOtRing()`
- `src/services/antiGravity/validators.ts` (85 linii) — walidacja `validateManhole()`

**Skutki:** Po usunięciu `selection.ts`, żaden inny plik w projekcie nie importuje z `prismaClient` przez antiGravity. `prismaClient.ts` i `prisma` pozostają w użyciu przez inne route'y.

### Krok 2: Usuń route productsStudnieAuto

```bash
git rm src/routes/productsStudnieAuto.ts
```

Usuwane endpointy (103 linii):
- `POST /api/products-studnie/auto-select` — wywołuje `selectManholeComponents()`
- `GET /api/products-studnie/available-components/:dn` — wywołuje `getAvailableComponents()`
- `POST /api/products-studnie/validate` — wywołuje `validateManhole()`

**Weryfikacja:** Frontend nie wywołuje żadnego z tych endpointów. Jedyna zewnętrzna zależność to Python ML na porcie 8000 (`wellSolver.js:238`).

| Błąd kompilacji TypeScript po usunięciu | Niskie | Uruchomić `npx tsc --noEmit` po zmianach |

## Weryfikacja po zmianach

1. **Kompilacja TypeScript:** `npx tsc --noEmit` — musi przejść (usuwa martwy kod, nic nie psuje)
2. **Testy:** `npm test` — testy są samodzielne, nie importują antiGravity, więc powinny przejść
3. **Lint:** `npm run lint` — sprawdzić brak ostrzeżeń o nieużywanych importach
4. **Frontend:** Otworzyć aplikację w przeglądarce, przejść przez kreator studni, sprawdzić czy auto-dobór działa przez Python backend
5. **Serwer:** Uruchomić `npm run dev` i sprawdzić czy serwer startuje bez błędów
