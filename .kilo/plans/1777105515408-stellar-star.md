# Plan: Analiza i Poprawa Aplikacji Studni - System Doboru Elementów

## 1. Podsumowanie Analizy

### 1.1 Architektura Systemu

System składa się z **DWU niezależnych solverów** dobierania elementów studni kanalizacyjnych:

| Komponent | Technologia | Lokalizacja | Port |
|-----------|-------------|-------------|------|
| **Backend AI** | Python + OR-Tools CP-SAT | `well_configurator_backend/` | 8000 |
| **Frontend JS** | Vanilla JS + DP Solver | `public/js/studnie/` | - (fallback) |

**Strategia działania (offline-first):**
1. Frontend wysyła zapytanie do backendu na `http://localhost:8000/api/v1/configure`
2. Jeśli backend niedostępny → używa lokalnego solvera JS (`runJsAutoSelection()`)
3. Jeśli backend zwróci `is_valid=True` → wynik jest akceptowany
4. Jeśli backend zwróci `is_valid=False` z elementami → fallback "Wymuszony"
5. Jeśli backend nie zwróci elementów → błąd

## 2. ZIDENTYFIKOWANE PROBLEMY

### 2.1 PROBLEMY KRYTYCZNE (muszą być naprawione)

#### 🔴 Problem 1: Niezgodność tolerancji między backendem a frontendem

**Backend** (`generator.py:79-104`):
```python
stages = [
    {"name": "Standard",   "tolerance_below": 60,  "tolerance_above": 20},
    {"name": "Optymalny",  "tolerance_below": 200, "tolerance_above": 20},
    {"name": "Ratunkowy",  "tolerance_below": 500, "tolerance_above": 20},
]
```

**Frontend** (`wellSolver.js:1058`):
```javascript
let solution = solve(280, 20, 280, false); // tolerance_below=280, tolerance_above=20
```

**Skutek:** Frontend może zaakceptować konfigurację, której backend by nie zaakceptował (lub odwrotnie). Różnice w tolerancji mogą prowadzić do niespójnych wyników między trybem online a offline.

**Lokalizacja kodu:**
- Backend: `generator.py:79-104`, `cp_optimizer.py:84-93`
- Frontend: `wellSolver.js:792-1056`, `ringOptimizer.js:21-53`

---

#### 🔴 Problem 2: Brak włazu w przypadku błędu backendu

**Opis:** W `finalizeSuccess()` (`wellSolver.js:329-362`) - jeśli backend zwróci `is_valid=True`, ale bez włazu, frontend automatycznie dodaje `WLAZ-150`. Jednak gdy backend zwróci `is_valid=False`, `finalizeSuccess` NIE jest wywoływane i studnia może pozostać bez włazu.

**Ścieżka błędu:**
1. Backend zwraca `is_valid=False` z `errors=["Brak dennic..."]`
2. Frontend ignoruje `items` i ustawia `well.config = []` (`wellSolver.js:380`)
3. Studnia ma pustą konfigurację → brak włazu

**Lokalizacja:** `wellSolver.js:364-384`

---

**Backend** (`rules.py:45-119`):
